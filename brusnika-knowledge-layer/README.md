## Блок 1: Pipeline Ингестии (Асинхронный / По триггеру)

Этот блок работает вне пользовательских запросов и отвечает за подготовку и загрузку качественных данных в векторную базу.

1. **Frontmatter Parser:** Чтение `.md` файлов, извлечение YAML-метаданных и их строгая валидация через Pydantic-модель `RawDocument`.
2. **Table Extractor & Summarizer:** Поиск таблиц по Regex, изоляция сырого Markdown в key-value хранилище, генерация текстового саммари через локальную Qwen3 и замена таблицы в тексте на плейсхолдер.
3. **MarkdownHeaderTextSplitter:** Иерархическая нарезка текста по заголовкам H1/H2/H3 с сохранением логической структуры.
4. **Contextual Chunks Builder:** Программное добавление пути заголовков (из сплиттера) непосредственно в начало текстового чанка перед его векторизацией. Это обогащает семантику чанка для модели эмбеддингов.
5. **Dual Vectorization:** Параллельная генерация двух типов векторов. Dense-векторы создаются через GigaEmbeddings / Qwen3-Embedding. Sparse-векторы (BM25/SPLADE) генерируются локально (например, через библиотеку `fastembed`).
6. **Vector Store (Qdrant):** Сохранение чанков в Qdrant, который нативно поддерживает мульти-векторный поиск. В БД улетает: Dense-вектор, Sparse-вектор и наш детализированный JSON-payload (доступы, домены, связи).

## Блок 2: Оркестратор запросов (Точка входа LangGraph)

Пользователь отправляет запрос. Инициализируется состояние графа (`State`), и в дело вступает локальная **Qwen3 (8GB VRAM)**.

1. **State Initialization:** Создание словаря состояний для LangGraph (содержит исходный запрос, историю, переписанный запрос, фильтры, список извлеченных чанков, флаги ошибок).
2. **Query Rewriter:** Локальная LLM анализирует запрос и историю сессии, разрешая анафору (заменяет местоимения на конкретные термины).
3. **Semantic Router:** Локальная LLM классифицирует намерение пользователя и выдает строгий JSON с параметрами: `domain` (домен знаний) и `access_level` (уровень доступа пользователя, пришедший из токена авторизации).

## Блок 3: Поиск и Фильтрация (Retrieval & Reranking)

Самый нагруженный логикой блок, объединяющий семантику, лексику и графовые связи.

1. **Hybrid Retriever:** Параллельный запрос в Qdrant. Dense-поиск ищет по смыслу, Sparse-поиск ищет по точным совпадениям ключевых слов. Оба запроса используют жесткие метадата-фильтры (домен, доступ), сформированные роутером.
2. **Reciprocal Rank Fusion (RRF):** Математическое объединение результатов Dense и Sparse поиска. Списки нормируются и сливаются в единый пул топ-кандидатов.
3. **Multi-hop Expansion:** Проверка поля `linked_docs` у найденных чанков. Если система видит сильную связь с другим документом, она делает дополнительный точечный запрос для подтягивания контекста.
4. **Cross-Encoder Reranker:** Легкая модель (например, `ms-marco-MiniLM` или локальный BGE-Reranker) берет топ-20 документов после RRF и заново оценивает их релевантность *именно к переписанному запросу пользователя*, оставляя топ-5 лучших.
5. **Context Grader:** Локальная Qwen3 проверяет оставшиеся чанки на статус (уведомляет, если документ в архиве) и оценивает их полезность. Если ни один чанк не релевантен, граф по `conditional_edge` уходит обратно на шаг `Query Rewriter` или просит пользователя уточнить вопрос.

## Блок 4: Генерация и Валидация (Когнитивный слой)

Сборка финального ответа. Здесь используется мощное API для сложного рассуждения.

1. **Parent-Child Resolver:** Замена плейсхолдеров таблиц (если в отфильтрованных чанках есть `table_summary`) на оригинальные Markdown-таблицы из хранилища.
2. **Generator (Gemini 2.5 Flash):** Тяжелая модель получает системный промпт, переписанный вопрос, собранный контекст и массив предупреждений (например, об устаревших статусах). Генерирует подробный ответ со ссылками на источники (`source_file`).
3. **Hallucination Checker:** Отдельный узел валидации. Локальная Qwen3 (или дешевый вызов API) получает связку «Сгенерированный ответ + Контекст из Retriever» и проверяет, не выдумала ли основная модель факты.
4. **Loop-on-Failure:** Условие LangGraph (`conditional_edge`). Если Hallucination Checker ловит галлюцинацию, запрос возвращается на этап генерации с указанием ошибки, либо система честно отвечает: «В базе знаний нет точной информации».
5. **Fallback:** Узел вежливого отказа, если на этапе Retriever/Grader не было найдено документов или сработала блокировка по доступам.

## Блок 5: Инфраструктура и Обвязка

Обеспечение стабильности и observability (наблюдаемости) системы.

1. **FastAPI Backend:** Асинхронное REST API для взаимодействия с фронтендом или Telegram-ботом.
2. **Session Manager:** Хранение истории сообщений (Redis или PostgreSQL) для обеспечения контекста диалога.
3. **Observability Pipeline:** Логирование метрик (время ответа локальных моделей vs API, затраты токенов, сработавшие фильтры Qdrant, количество попыток на этапе Hallucination Checker).
