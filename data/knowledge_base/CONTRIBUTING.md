# Руководство по написанию документов базы знаний

Правила оформления и публикации материалов в корпоративной базе знаний «БРУСНИКА».

## Структура файла

Каждый документ — файл Markdown (`.md`) со следующей структурой:

1. **YAML frontmatter** (обязателен, в начале файла)
2. **Заголовок H1** — совпадает с полем `title`
3. **Краткое описание** — 2–3 предложения о содержании и целевой аудитории
4. **Основное содержание** — 300–800 слов, заголовки H2 и H3
5. **Практический элемент** — процесс, таблица, чек-лист или пример
6. **Ответственные и контакты**
7. **Связанные документы**

## Frontmatter

```yaml
---
title: Название документа
description: Краткое описание в одно-два предложения.
domain: construction
audience:
  - employees
  - construction-site
access: internal
owner: "@Отдел ПТО"
last_reviewed: 2025-03-15
status: active
tags:
  - строительство
  - ПТО
related:
  - construction/pto-documentation.md
version: "1.2"
---
```

### Обязательные поля

| Поле | Допустимые значения |
|------|---------------------|
| `domain` | `development`, `construction`, `commercial`, `finance`, `hr`, `legal`, `it`, `exploitation`, `company` |
| `audience` | `employees`, `construction-site`, `management`, `sales`, `clients`, `partners` |
| `access` | `public`, `internal`, `confidential`, `restricted` |
| `status` | `draft`, `active`, `needs-review`, `archived` |

### Владелец (`owner`)

Указывайте фамилию ответственного или отдел со знаком `@`: `@Казначейство`, `@Иванова`.

## Стиль написания

- **Деловой русский** без канцелярита и штампов.
- **Конкретика:** «в течение 3 рабочих дней», а не «в разумные сроки».
- **Роли в процессах:** кто инициирует, кто согласует, кто утверждает.
- **Цифры и пороги:** суммы, сроки, KPI — с конкретными значениями.
- **Терминология:** специализированные термины (РНС, ППР, ДДУ) используются по контексту — с расшифровкой в обзорных документах, без расшифровки в профильных.

## Именование файлов

- Только латиница, строчные буквы, дефисы: `land-acquisition-process.md`
- Путь отражает домен: `construction/procurement-process.md`

## Процесс публикации

1. Автор готовит черновик со статусом `draft`.
2. Владелец процесса (`owner`) проводит ревью.
3. После согласования статус меняется на `active`, обновляется `last_reviewed`.
4. Ежегодный пересмотр: документы без обновления более 12 месяцев получают статус `needs-review`.
5. Устаревшие регламенты архивируются (`archived`) с пометкой о замене.

## Связи между документами

- Поле `related` в frontmatter — машиночитаемые связи.
- Раздел «Связанные документы» в теле — человекочитаемые ссылки.
- При изменении процесса проверяйте все документы из `related` на согласованность.

## Чего избегать

- Дублирования полного текста процесса в нескольких файлах (допустимо частичное пересечение с отсылкой к основному регламенту).
- Противоречивых порогов и сроков между документами одного домена (согласовывайте с владельцем).
- Конфиденциальных данных в документах с `access: internal` или `public`.

## Тестовые вопросы для RAG

База знаний сопровождается двумя файлами с тестовыми вопросами:

| Файл | Назначение |
|------|------------|
| `test-questions.md` | Вопросы для проверки RAG — **передаётся** вместе с базой знаний |
| `Тестовые вопросы (не для отправки).md` | Полный набор с критериями оценки — **не передаётся** |

При подготовке комплекта для кандидата:

1. Включите `test-questions.md` и все папки с регламентами.
2. **Исключите** `Тестовые вопросы (не для отправки).md` — в нём резервные вопросы для финальной проверки и эталонные ответы.
3. При изменении вопросов обновляйте оба файла и таблицу соответствия в полном наборе.

Критерии оценки (тип, источники, ожидаемое поведение) ведутся **только** во внутреннем файле.
