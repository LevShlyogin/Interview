# Архитектурная записка: Knowledge Layer

Данный документ описывает архитектуру RAG-системы, спроектированной для работы с внутренней базой знаний девелоперской компании. Архитектура построена на паттерне **Agentic RAG (Hybrid Routing & Validation)**, что позволяет минимизировать галлюцинации и обеспечить строгий контроль доступов.

## 1. Пайплайн Ингестии (Подготовка данных)

Этот блок отвечает за подготовку и загрузку качественных данных в векторную базу.

1. **Frontmatter Parser:** Чтение `.md` файлов, извлечение YAML-метаданных и нормализация под единую Pydantic-подобную схему (RawDocument).
2. **Table Extractor & Summarizer:** Поиск Markdown-таблиц по Regex, их извлечение и замена на плейсхолдеры. Локальная модель (Qwen 2.5 3B) генерирует текстовое саммари таблицы, которое векторизуется вместо сырой разметки.
3. **Smart Markdown Splitter:** Иерархическая нарезка текста по заголовкам. **Обоснование решения:** Мы намеренно ограничили сплиттер нарезкой только по H1 и H2, игнорируя H3. Это решило проблему "фрагментации контекста", позволив связанным терминам (ПОС, ППР) оставаться в одном семантическом чанке.
4. **Contextual Builder:** Программное добавление пути заголовков (например, `[Header 1 -> Header 2]`) в начало чанка для обогащения семантики.
5. **Dual Vectorization:** Генерация двух типов векторов локально через `fastembed`.
* *Dense-векторы:* `paraphrase-multilingual-MiniLM-L12-v2` для поиска по смыслу.
* *Sparse-векторы:* `Qdrant/bm25`. **Обоснование:** Был выбран классический алгоритм BM25 вместо нейросетевого SPLADE, так как BM25 работает надежнее при поиске специфичных строительных русскоязычных аббревиатур (КСГ, СМР, ИД).


6. **Vector Store:** Данные загружаются в локальный Qdrant с настроенными Payload-индексами для фильтрации по доменам и доступам.

## 2. Оркестратор запросов (Точка входа LangGraph)

Управление потоком выполнения (State Graph). Для легких и быстрых задач классификации используется локальная **Qwen 2.5 3B (Ollama)**.

1. **Query Rewriter:** Локальная LLM анализирует вопрос пользователя с учетом истории сессии и переписывает его в самостоятельный запрос (разрешение анафоры).
2. **Semantic Router:** Классификация намерения пользователя с выдачей структурированного JSON. Роутер определяет целевой `domain` поиска.
3. **Smart Fallback:** Если данные не найдены или запрос размыт, локальная модель генерирует вежливый уточняющий вопрос (например, *"Вас интересует регламент для коммерческой или жилой недвижимости?"*).

## 3. Поиск и Фильтрация (Retrieval & Reranking)

Блок, объединяющий семантику, точный лексический поиск и бизнес-правила.

1. **Hybrid Retriever:** Параллельный запрос в Qdrant (Dense + BM25). В запросе используется жесткая Payload-фильтрация по `domain` (найденный домен + глобальный справочник `company`) и `access_level` пользователя.
2. **Reciprocal Rank Fusion (RRF):** Математическое объединение результатов Dense и Sparse поиска средствами Qdrant.
3. **Cross-Encoder Reranker:** Модель `DiTy/cross-encoder-russian-msmarco` берет широкий пул кандидатов и переранжирует их относительно переписанного запроса, решая проблему "Lost in the Middle".
4. **Rule-Based Grader:** Детерминированный узел, который проверяет метаданные найденных чанков (например, `status == "archived"`) и формирует массив предупреждений для генератора об устаревшей информации.

## 4. Генерация и Валидация (Когнитивный слой)

Для финальной сборки ответа используется мощная облачная LLM, работающая под строгим контролем.

1. **Generator:** Тяжелая модель получает жесткий системный промпт (с позитивными инструкциями), контекст и массив предупреждений. Модель обучена выявлять противоречия (разные цифры/сроки в разных файлах) и открыто сообщать о них пользователю.
2. **Deterministic Validator (Post-Processing):** Внедрен Regex-валидатор, проверяющий ответ модели.
* Анонимизирует пути файлов (чтобы не отдавать внутренний нейминг).
* Вырезает галлюцинированные призывы к действию (CTAs, Markdown-ссылки).
3. **Self-Correction Loop:** При провале валидации (например, сгенерирован подозрительно короткий текст) граф уходит на повторную генерацию (Retry) с указанием конкретной ошибки.

## 5. Планы развития (Что было бы сделано при наличии времени и ресурсов)

Для вывода системы в полноценный Production потребовались бы следующие доработки:

1. **Multi-hop Expansion (Graph RAG):** Сейчас поле `linked_docs` сохраняется в БД, но не используется. При наличии времени был бы добавлен механизм, который делает дополнительный точечный запрос в Qdrant для подтягивания связанных документов перед генерацией ответа.
2. **Parent-Child Table Resolver:** Вместо передачи LLM-генератору текстового саммари таблицы, система подтягивала бы оригинальную Markdown-разметку таблицы из Key-Value хранилища по `table_id` для более детального ответа.
3. **LLM-based Hallucination Checker:** Расширение текущего детерминированного валидатора модулем LLM-as-a-judge для глубокой проверки на фактологические искажения (Self-RAG паттерн).
4. **Инфраструктурная обвязка:** Перенос пользовательского интерфейса со Streamlit на асинхронный FastAPI backend + Vue.js/React frontend, а также внедрение Redis для персистентного хранения сессий и LangSmith/Phoenix для Observability (логирование метрик и затрат токенов).
5. **Инфраструктурная обвязка:** Перенос пользовательского интерфейса со Streamlit на асинхронный FastAPI backend (создание полноценного REST API для чата) + Vue.js/React frontend. Внедрение Redis для высокоскоростного кэширования сообщений пользователя и персистентного хранения сессий. Для Observability мониторинг метрик, затрат токенов и трейсинг.
6. **Production Engineering & QA:** Внедрение лучших практик промышленной разработки. Покрытие графа и пайплайна ингестии Unit- и E2E-тестами, настройка комплексной обработки исключений и сквозного логирования (ELK/Graylog). Также потребуется системный benchmarking пайплайна моделей — от подбора оптимальных эмбеддеров и реранкеров, до генератора под специфику компании.
7. **Human-in-the-Loop (HITL) & Semantic Caching:** Внедрение системы пользовательской оценки ответов (например, по 10-балльной шкале). Пары «Вопрос-Ответ», получившие высокую оценку (≥ 7), автоматически маршрутизируются в отдельную базу данных (векторный кэш эталонных ответов). При поступлении аналогичного запроса система сначала ищет совпадение в кэше: это радикально снижает latency, экономит токены тяжелой LLM и гарантирует выдачу проверенного фактологического ответа. Кроме того, собранный массив высокооцененных пар формирует золотой датасет для будущего.

## 6. Стратегия автоматического обновления индекса (CI/CD)

Для обеспечения актуальности базы знаний при изменениях Markdown-файлов в репозитории (GitHub/GitLab), я бы реализовал событийную архитектуру (Event-Driven Pipeline):

1. **Webhook Trigger:** При каждом `git push` или `merge` в главную ветку репозитория с базой знаний, GitHub Actions отправляет Webhook на эндпоинт нашего сервиса ингестии.
2. **Delta Processing (Обработка дельты):** В payload вебхука передается список измененных, добавленных и удаленных файлов. Скрипт не пересобирает всю базу (что дорого и долго), а работает только с дельтой.
3. **Удаление по Source_file:** Для измененных и удаленных файлов скрипт делает запрос в Qdrant на удаление всех пойнтов, у которых `payload.source_file == filename`.
4. **Парсинг и Векторизация:** Обновленные и новые файлы проходят пайплайн `Parser -> Extractor -> Splitter -> Vectorizer` и загружаются в Qdrant (Upsert).
5. **Zero-Downtime:** Для крупных обновлений использовался бы паттерн *Collection Aliases* в Qdrant (сборка индекса в теневую коллекцию и мгновенное переключение алиаса), чтобы поиск не прерывался ни на секунду.

Если база знаний лежит не в репозитории, мы больше не можем полагаться на CI/CD (GitHub Actions). Нам нужно строить **ETL-пайплайн (Extract, Transform, Load)**. Архитектурная стратегия в этом случае разветвляется на три подхода в зависимости от "современности" источника данных.

---

### Стратегия 1: Webhooks (Event-Driven / Push-модель)

*Подходит для: Confluence, Notion, Bitrix24, современных CMS.*

Это самый идеальный сценарий, максимально близкий к нашему `git push`.

1. **Настройка источника:** В админке Confluence или Notion мы настраиваем Webhook на события `page_created`, `page_updated`, `page_deleted`.
2. **Слушатель (Listener):** На нашем FastAPI бэкенде мы поднимаем отдельный эндпоинт (например, `/api/webhooks/confluence`).
3. **Обработка дельты:** * Сотрудник нажимает "Сохранить" в Confluence.
* Confluence мгновенно шлет нам JSON с ID страницы и новым текстом (или ссылкой на API для скачивания).
* Мы парсим HTML/XML в наш стандартный текст, удаляем из Qdrant старые векторы по `page_id` и векторизуем новые.
* Индекс обновляется в реальном времени.



### Стратегия 2: Scheduled Polling (Pull-модель по расписанию)

*Подходит для: SharePoint, Google Drive, Jira, систем с REST API, но без нормальных вебхуков.*

Здесь инициатором выступает наша система, а не источник данных. Нам понадобится оркестратор задач (например, **Apache Airflow**, **Celery Beat** или **n8n**).

1. **Трекинг состояния (Sync DB):** Нам придется завести небольшую реляционную базу (PostgreSQL или SQLite), чтобы помнить, что и когда мы скачали. Таблица `documents_sync_state`: `doc_id`, `last_modified_at`.
2. **Крон-джоба:** Раз в час (или каждую ночь) Airflow запускает скрипт.
3. **Запрос дельты:** Скрипт делает легкий API-запрос к источнику: *"Дай мне список всех файлов, у которых `updated_at` больше, чем время моей последней проверки"*.
4. **Изолированное обновление:** Система скачивает только те 5-10 файлов, которые изменились за день, прогоняет их через наш `Splitter` -> `Vectorizer` и делает Upsert в Qdrant.

### Стратегия 3: Document Hashing (Слепая синхронизация)

*Подходит для: Сетевых папок (SMB), локальных дисков, старых ERP-систем без API.*

Самый "тяжелый", но часто неизбежный сценарий для старых компаний. Система не знает, когда файл изменился, потому что у файлов кривые или недостоверные метаданные (даты изменения).

1. **Сканирование:** Скрипт (например, раз в сутки) рекурсивно обходит всю сетевую папку и читает содержимое всех файлов (PDF, Docx).
2. **MD5 / SHA-256 Хеширование:** Для каждого файла вычисляется криптографический хеш от его текста.
3. **Сравнение:** Скрипт сравнивает свежий хеш с хешем, который лежит в нашей базе `Sync DB`.
* Хеш совпадает -> пропускаем файл (экономим деньги на векторизации).
* Хеш изменился -> файл отредактировали. Удаляем старые чанки из Qdrant, векторизуем заново.
* Файла больше нет на диске, а хеш в базе остался -> файл удален. Вычищаем его векторы из Qdrant.

### Архитектурные требования к самому пайплайну

Независимо от того, какую стратегию мы выберем, ядро нашего кода (начиная с шага нарезки на чанки) останется прежним. Однако нам придется добавить два новых микросервиса:

1. **Connector Layer (Слой коннекторов):** Нам придется написать или использовать готовые библиотеки (например, `LangChain Document Loaders` или `LlamaIndex Data Connectors`) для конвертации проприетарных форматов (.docx, .pdf, Confluence HTML) в чистый текст, с которым умеет работать наш `HierarchicalSplitter`.
2. **Dead Letter Queue (Очередь ошибок):** Если Confluence временно недоступен или PDF-файл оказался битым (сканированным без OCR), пайплайн не должен падать. Ошибочный документ должен откладываться в специальную очередь (DLQ) для ручного разбора или автоматического ретрая (повторной попытки), а остальные документы должны успешно загрузиться в базу.
